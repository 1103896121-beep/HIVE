# 代码审查完整清单

此文档提供详细的代码审查检查项，用于确保代码质量、安全性和性能。

---

## 📝 变更简介 (Summary)

**审查前必填：**

- PR 标题：简洁描述变更内容
- 变更类型：[新功能 / Bug 修复 / 重构 / 性能优化 / 文档更新]
- 影响范围：列出受影响的模块/组件
- 相关链接：Jira/Issue 编号、设计文档链接

---

## ✅ 自查清单 (Self-Check before Requesting Review)

**在指派 Reviewer 之前，请确保已经完成了以下检查：**

### 1. 基础检查

- [ ] **代码可以通过编译/运行**，且本地测试通过
- [ ] **所有的 CI/CD 流水线**（Lint, Unit Test）都已通过
- [ ] **没有提交调试代码**（如 `console.log`, `print`, `debugger`）或注释掉的代码块
- [ ] **PR 的粒度适中**（建议不超过 400 行变更），且专注于单一目标
- [ ] **删除了无用代码**：未使用的导入、变量、函数
- [ ] **代码格式统一**：使用了项目的格式化工具（Prettier/Black）

### 2. 功能与逻辑 (Functionality)

- [ ] **代码实现了需求描述的所有功能**
- [ ] **边界情况** (Edge Cases) 已处理，包括：
  - [ ] 空值、null、undefined
  - [ ] 负数、零、极大值
  - [ ] 空数组、空对象、空字符串
  - [ ] 超长字符串或数组
  - [ ] 网络异常、超时
  - [ ] 并发访问
- [ ] **是否存在并发问题**（Race Conditions）？
  - [ ] 异步操作顺序是否正确
  - [ ] 是否有资源竞争
- [ ] **错误处理完善**：
  - [ ] 异常日志清晰且包含上下文
  - [ ] 没有吞掉异常（空 catch 块）
  - [ ] Python：避免裸 `except`
  - [ ] 前端：有友好的错误提示

### 3. 代码质量与可读性 (Quality & Readability)

- [ ] **命名规范**：
  - [ ] 变量、函数名清晰易懂
  - [ ] 避免了 `temp`, `data`, `info`, `item` 等模糊命名
  - [ ] 布尔值使用 `is/has/should` 前缀
  - [ ] 遵循项目命名约定：
    - 变量/函数：camelCase
    - 类/组件：PascalCase
    - 常量：UPPER_SNAKE_CASE
- [ ] **DRY 原则**（Don't Repeat Yourself）：
  - [ ] 没有重复代码
  - [ ] 通用逻辑已抽离为函数/组件
  - [ ] 魔法数字已提取为常量
- [ ] **KISS 原则**（Keep It Simple, Stupid）：
  - [ ] 代码逻辑简单直接
  - [ ] 没有过度设计
  - [ ] 函数职责单一
  - [ ] 单个函数不超过 50 行（建议）
- [ ] **注释质量**：
  - [ ] 复杂逻辑有必要的注释
  - [ ] 注释解释 "为什么" 这么做，而不是 "做了什么"
  - [ ] 使用标记：TODO、FIXME、NOTE、HACK
  - [ ] 删除了过时的注释
- [ ] **代码结构清晰**：
  - [ ] 相关代码放在一起
  - [ ] 依赖关系清晰
  - [ ] 模块划分合理

### 4. 测试 (Tests)

- [ ] **核心逻辑补充了对应的单元测试**
- [ ] **测试覆盖了"快乐路径"** (Happy Path)
- [ ] **测试覆盖了"异常路径"** (Failure Cases)：
  - [ ] 输入验证失败
  - [ ] 网络错误
  - [ ] 数据库错误
  - [ ] 权限不足
- [ ] **测试代码本身清晰易读**
- [ ] **测试数据合理**：
  - [ ] 使用了有意义的测试数据
  - [ ] 避免了硬编码的 ID 或日期
- [ ] **测试独立性**：
  - [ ] 测试之间不相互依赖
  - [ ] 每个测试可以独立运行

### 5. 安全与性能 (Security & Performance)

#### 安全性

- [ ] **没有硬编码敏感信息**：
  - [ ] 密码、Token、API 密钥
  - [ ] 数据库连接串
  - [ ] 加密密钥
  - [ ] 敏感信息从环境变量读取
- [ ] **输入数据进行了校验和清洗**：
  - [ ] 前端：使用 Schema 验证（如 Zod）
  - [ ] 后端：使用 Pydantic 验证
  - [ ] 防止 SQL 注入（使用参数化查询）
  - [ ] 防止 XSS（避免 dangerouslySetInnerHTML）
  - [ ] 防止 CSRF（使用 Token）
- [ ] **权限校验**：
  - [ ] 后端：权限检查在 service 层完成
  - [ ] 前端：敏感功能有权限判断
- [ ] **敏感数据脱敏**：
  - [ ] 日志中不包含敏感信息
  - [ ] API 响应中敏感字段已脱敏
  - [ ] 密码等数据已加密存储

#### 性能

- [ ] **数据库查询已优化**：
  - [ ] 避免了 N+1 查询
  - [ ] 使用了适当的索引
  - [ ] 避免了 SELECT \*
  - [ ] 大数据量使用分页
- [ ] **前端性能**：
  - [ ] 避免了不必要的重渲染
  - [ ] 合理使用 useMemo/useCallback
  - [ ] 列表渲染使用了稳定的 key
  - [ ] 大列表使用虚拟滚动
  - [ ] 路由与组件支持懒加载
- [ ] **资源使用合理**：
  - [ ] 没有内存泄漏（事件监听器已清理）
  - [ ] 文件/连接使用后已关闭
  - [ ] 避免了无限循环
- [ ] **网络请求优化**：
  - [ ] 避免了重复请求
  - [ ] 使用了适当的缓存
  - [ ] 批量操作代替多次单独请求

## 🎯 Reviewer 指南

作为 Reviewer，除了检查上述清单外，还应注意：

1. **架构一致性**：变更是否符合项目整体架构
2. **代码风格一致性**：是否与现有代码风格保持一致
3. **向后兼容性**：是否破坏了现有 API 或功能
4. **文档更新**：是否需要更新相关文档
5. **依赖管理**：新增依赖是否必要且安全

---

## 📋 常见问题检查

### 前端常见问题

- [ ] 状态管理是否合理（避免 prop drilling）
- [ ] 是否滥用 useEffect
- [ ] 异步操作是否处理了加载和错误状态
- [ ] 表单是否有验证和错误提示
- [ ] 是否有适当的 loading 状态

### 后端常见问题

- [ ] 是否在 api 层直接操作数据库（违反分层架构）
- [ ] 是否使用了 print 而不是 logging
- [ ] 异步函数是否正确使用 async/await
- [ ] 事务处理是否正确
- [ ] API 是否遵循 RESTful 规范

---

## ✨ 最佳实践提醒

1. **小步提交**：一次 PR 只做一件事
2. **先重构再添加功能**：如果需要重构，单独提一个 PR
3. **及时回应评审意见**：积极讨论，解释设计决策
4. **自我审查**：提交前自己先完整走一遍清单
5. **持续学习**：从评审意见中学习改进
