# Code Review Skill

此 skill 提供系统化的代码审查流程和完整的审查清单，帮助确保代码质量、可读性、安全性和性能。

## 使用场景

在以下情况下使用此 skill：

- 用户请求审查代码或 Pull Request
- 用户需要生成代码审查清单或 PR 模板
- 用户询问代码审查最佳实践
- 在提交代码前需要进行自查

## 审查流程

### 1. 理解变更范围

首先了解代码变更的目的和范围：

- 阅读相关的需求描述、Issue 或设计文档
- 确定变更影响的模块和组件
- 识别潜在的影响范围

### 2. 执行分层审查

按照以下优先级顺序执行审查：

**第一优先级：功能正确性与安全性**

- 功能是否符合需求
- 边界情况是否处理
- 安全漏洞检查（输入验证、敏感信息）

**第二优先级：代码质量**

- 命名规范
- 逻辑清晰度
- 代码复用（DRY 原则）

**第三优先级：测试与性能**

- 测试覆盖率
- 性能优化建议

### 3. 使用详细清单

参考 `references/checklist.md` 中的完整审查清单，确保不遗漏任何重要检查项。该清单包含：

- 基础检查（编译、CI/CD、调试代码清理）
- 功能与逻辑（需求实现、异常处理、边界情况）
- 代码质量与可读性（命名、DRY/KISS 原则、注释）
- 测试（单元测试、覆盖率）
- 安全与性能（敏感信息、数据库优化、输入校验）

### 4. 生成审查报告

根据审查结果，生成结构化的报告：

```markdown
## 审查摘要

- 总体评价：[通过/需修改/存在严重问题]
- 主要发现：[列出关键问题]

## 详细发现

### 🔴 严重问题（必须修复）

1. [问题描述 + 位置 + 建议]

### 🟡 建议改进（可选）

1. [改进建议 + 理由]

### ✅ 优点

1. [值得肯定的地方]

## 检查清单

- [x] 功能正确性
- [x] 安全性检查
- ...
```

## 技术规范参考

审查时应遵循项目的技术规范：

**前端（React + TypeScript）**

- 使用函数组件和 Hooks
- Props 必须有 TypeScript 类型定义
- 避免不必要的重渲染
- 命名：组件用 PascalCase，函数/变量用 camelCase

**后端（Python）**

- Python >= 3.10
- 函数必须标注类型
- 分层架构：api → service → repository
- 使用 logging 模块，不使用 print

**通用规范**

- 注释解释"为什么"，不解释"是什么"
- 禁止硬编码敏感信息
- 错误处理必须完善
